## 1. Описание проекта

В рамках лабораторной работы был доработан мой тг To Do бот, предназначенный для управления личными задачами. Бот позволяет пользователям создавать задачи, распределять их по приоритетам и категориям, устанавливать напоминания, а также отслеживать статус выполнения

Основная цель лабораторной заключалась в улучшении простого и функционального инструмента для управления задачами с возможностью дальнейшего расширения на основе пользовательского фидбека

Ссылка на бота: [https://t.me/itmoftmi_vinogradskaianl_bot](https://t.me/itmoftmi_vinogradskaianl_bot)

## 2. Выбор платформы для деплоя

Для развертывания приложения был выбран сервис Railway.

Причины выбора:

* автоматическая интеграция с GitHub
* простой процесс деплоя без ручной настройки сервера
* поддержка переменных окружения
* возможность автоматического перезапуска при обновлениях кода

Данный вариант был выбран как наиболее быстрый и стабильный для учебного проекта без необходимости настройки инфраструктуры вручную.

## 4. Проблемы при деплое и их решение

### 1) Конфликт getUpdates

Проблема возникала из-за одновременного запуска нескольких экземпляров бота. Telegram API блокировал повторные подключения

* оставлен только один активный деплой в Railway
* локальные запуски были отключены

---

### 2) Ошибка webhook 

Telegram фиксировал активный webhook, из-за чего polling не запускался корректно

* выполнен вызов deleteWebhook через API Telegram
* проверено через getWebhookInfo

---

### 3) Ошибка InvalidToken

Причина заключалась в некорректной или устаревшей переменной BOT_TOKEN в окружении Railway

* создан новый бот через BotFather
* обновлен токен в Railway Variables
* выполнена повторная проверка загрузки переменной окружения

---

### 4) Ошибка JobQueue

При добавлении системы напоминаний возникла ошибка отсутствия job_queue

* не была установлена дополнительная зависимость python-telegram-bot[job-queue]

в итоге:
* добавлена зависимость в requirements.txt
* пересобран проект в Railway

## 5. Переход на базу данных SQLite

Изначально данные хранились в упрощенной форме, без долговременного хранения.

После анализа требований была внедрена SQLite база данных.

### Причины перехода:

* необходимость сохранения задач между перезапусками бота
* поддержка нескольких пользователей через user_id
* расширение функционала (статусы, теги, напоминания)

### Реализация:

Была создана таблица tasks со следующими полями:

* id
* user_id
* text
* priority
* tag
* reminder_time
* is_done
* is_deleted
* created_at

Добавлены функции:

* add_task
* get_tasks
* mark_done
* delete_task
